Chapter 01

What is Git?

Git is a distributed version control system. Think of it as a time machine for your code — you can save snapshots, go back in time, and work on multiple versions simultaneously.

📸

Snapshots, not diffs

Git stores complete snapshots of your project at each save point. If a file hasn't changed, Git just links to the previous version rather than storing it again.

🌐

Distributed

Every developer has a full copy of the repository, including all history. You can work completely offline and sync later.

🌿

Branching is cheap

Creating branches is instant. You can experiment freely without affecting anyone else, then merge your work back when ready.

🔒

Integrity guaranteed

Every file and commit is checksummed with SHA-1. It's impossible to change history without Git knowing about it.

Chapter 02

The Git Workflow

Git has four key areas where your files live. Understanding how files move between them is the foundation of Git mastery.

// the four areas of git
📝
Working Directory
Your actual files on disk
git add → rm --cached
📋
Staging Area
Files ready to commit
git commit → git reset
🗄️
Local Repo
Your commit history
git push → git pull
☁️
Remote Repo
GitHub / GitLab etc.
1

$ git init — Create a repository

Initializes a new Git repo in your current folder. Creates a hidden .git directory that stores all the history.

2

$ git add — Stage your changes

Move files from your working directory to the staging area, telling Git "I want these changes in my next commit."

3

$ git commit — Save a snapshot

Creates a permanent snapshot of staged files with a message describing what changed. This is your save point.

4

$ git push — Share your work

Upload your local commits to a remote repository so others can see and collaborate on your work.

Commits & History

A commit is a snapshot of your entire project at a moment in time. Each one links back to its parent, forming a chain of history.

HEAD is simply a pointer to the commit you're currently working on. It usually points to the tip of your current branch. When you make a new commit, HEAD moves forward automatically.
git log View commit history. Shows author, date, and message for each commit.
git log --oneline Compact history view. One line per commit — great for seeing the big picture.
git show a1b2 Inspect a commit. Shows what changed in that exact snapshot.
git diff See uncommitted changes. What's different between your files and the last commit.

Chapter 04

The Staging Area

The staging area (or "index") is Git's superpower. It lets you craft precise commits — selecting exactly which changes belong together, even from within the same file.

// working directory → staging → commit
Working Directory
index.html modified
styles.css new
old.js deleted
app.js modified
git add index.html git add styles.css ⟶

Staging Area

index.html staged
styles.css staged
git commit -m "update UI"
Only staged files go into the commit. app.js and old.js remain unchanged.
git status See what's changed. Shows staged, unstaged, and untracked files at a glance.
git add . Stage everything. Adds all changed files in the current directory.
git add -p Stage hunks interactively. Go through each change and decide what to stage. Powerful!
git restore --staged Unstage a file. Removes it from staging area but keeps your changes in the working directory. 
$ git status
On branch main

Changes to be committed (staged):
  modified:   index.html
  new file:   styles.css

Changes not staged:
  modified:   app.js

Untracked files:
  old_backup.js
Chapter 05

Branches

A branch is just a lightweight pointer to a commit. Creating one is instant. It lets you diverge from the main line of development and experiment safely.

// branch visualization — feature branch diverges from main
git branch feature Create a new branch. Makes a new pointer at the current commit without switching to it.
git switch feature Switch branches. Moves HEAD to the feature branch. Your files update to match.
git switch -c feature Create and switch. The shortcut — creates and immediately switches to the new branch.
git branch -d feature Delete a branch. Safe delete — only works if the branch is fully merged.
💡 Branch naming tip: Use descriptive names like feature/user-auth, fix/login-bug, or chore/update-deps — your future self will thank you.
Chapter 06

Merge & Rebase

When your branch work is done, you need to integrate it back. Git gives you two powerful ways: merge (preserves history) and rebase (rewrites history for a cleaner line).

// merge vs rebase — two ways to integrate branches

git merge — Preserves History

Creates a new "merge commit" that ties both branches together. Full history is preserved.

git rebase — Linear History

Replays your commits on top of the target branch. Cleaner, linear history — but rewrites commit IDs.
git merge feature Merge feature into current branch. Safe for shared branches. Creates a merge commit.
git rebase main Rebase onto main. While on feature branch, replays your commits on top of the latest main.
git merge --no-ff Always create merge commit. Even for fast-forward merges — useful for tracking feature integration points.
⚠️ Golden Rule of Rebase: Never rebase commits that have been pushed to a shared remote branch. Rebasing rewrites history — others who have the original commits will have conflicts.
Chapter 07

Working with Remotes

Remotes are versions of your repository hosted on the internet (GitHub, GitLab, etc.). They enable collaboration, backup, and deployment.

// local ↔ remote communication
💻

Local Repository

Your machine

main
feature
git push →
← git pull
← git fetch
☁️

Remote (origin)

GitHub / GitLab

origin/main
fetch vs pull: git fetch downloads changes from remote but doesn't apply them — your files stay the same. git pull = fetch + merge in one step. Prefer fetch when you want to review changes before merging.
git clone <url> Download a repo. Creates a local copy of a remote repository, including all history.
git remote -v List remotes. Shows the name and URL of all configured remote repositories.
git push -u origin main Push and track. Pushes to origin/main and sets it as the default upstream for future pushes.
git fetch --all Fetch all remotes. Downloads all branches from all remotes without changing your files.
git pull --rebase Pull with rebase. Instead of creating a merge commit, replays your local commits on top of remote changes.
Chapter 08

Command Cheatsheet

Every essential Git command, organized by category. Keep this handy.

🚀 Setup & Init

git init             # new repo
git clone <url>      # clone existing
git config --global  # global settings

📸 Save Changes

git add .            # stage all
git add -p           # interactive stage
git commit -m "msg"  # commit
git commit --amend   # fix last commit

🌿 Branches

git branch           # list branches
git switch -c feat   # create+switch
git merge feat       # merge branch
git rebase main      # rebase

☁️ Remote

git fetch            # download
git pull             # fetch + merge
git push             # upload commits
git remote -v        # list remotes

🔍 Inspect

git status           # what changed
git log --oneline    # history
git diff             # see changes
git blame <file>     # who changed what

⏮ Undo

git restore <file>   # discard changes
git reset HEAD~1     # undo last commit
git revert <hash>   # safe undo
git stash            # shelve changes
